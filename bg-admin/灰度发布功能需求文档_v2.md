## 一、需求背景

当前平台新功能上线时，如果直接对全部会员开放，一旦功能存在异常，可能影响所有用户使用，尤其是充值、投注、活动、个人中心等核心模块，风险更高。

因此需要建设一套**功能级灰度发布能力**，支持新功能先投放给部分会员使用，验证功能稳定、数据正常后，再单独推送全量。

本次灰度发布**不做环境隔离**，灰度用户和正式用户使用同一套正式数据。灰度只控制功能是否展示、入口是否开放、流程是否走新版逻辑，不单独拆数据库、不单独拆会员、不单独拆订单。

---

## 二、需求目标

1. 支持功能级灰度发布。
2. 支持多个功能同时灰度。
3. 每个灰度功能独立配置投放人群。
4. 每个灰度功能可单独推全量。
5. 每个灰度功能可单独关闭回滚。
6. 支持指定会员、VIP 等级、会员标签、注册时间、百分比放量等投放规则。
7. 支持同一会员命中多个灰度功能。
8. 支持互斥组，解决同一页面或同一流程多个灰度版本冲突问题。
9. 会员端根据后端返回的功能 Key 判断展示新版或旧版。
10. 灰度数据与正式数据互通，不做环境隔离。
11. 后台记录所有关键操作日志，便于问题追踪。
12. 灰度结果对同一会员稳定，放量比例提升时不剔除原命中会员。
13. 涉及资金链路的数据可追溯产生版本，回滚后历史数据可正常查询与处理。

## 三、名词定义

### 3.1 灰度发布

灰度发布是指新功能上线时，不直接对全部会员开放，而是先开放给部分会员使用。通过观察功能稳定性、业务数据、用户反馈后，再决定是否继续扩大范围、推送全量或关闭回滚。

本需求中的灰度发布只控制功能是否可见或是否走新版逻辑，不做独立环境隔离。

### 3.2 功能级灰度

功能级灰度是指以单个功能为单位进行灰度控制，而不是以整个版本为单位控制。

例如：

```text
新版充值页是一个灰度功能
新版活动中心是一个灰度功能
新版投注确认流程是一个灰度功能
```

每个功能都有独立配置，可以单独灰度、单独推全量、单独关闭。

### 3.3 功能 Key

功能 Key 是灰度功能的唯一标识，前端和后端通过该 Key 判断会员是否命中某个灰度功能。

示例：

```text
new_recharge_page
new_activity_center
new_bet_confirm_flow
```

要求：

```text
功能 Key 全局唯一
创建后不建议修改
前端、后端、测试统一使用同一个 Key
```

### 3.4 灰度功能

灰度功能是指被纳入灰度控制的新功能、新页面、新入口、新流程或新交互。

例如：

```text
新版充值页
新版活动中心
新版投注确认弹窗
新版个人中心入口
```

### 3.5 灰度规则 / 投放规则

灰度规则是后台配置的投放条件，用于判断哪些会员可以看到新功能。

本期支持的规则包括：

```text
指定会员白名单
VIP 等级
会员标签
注册时间段
百分比放量
```

会员满足配置规则后，即可命中对应灰度功能。

### 3.6 命中灰度

命中灰度是指会员满足某个灰度功能的投放规则，后端判断该会员可以使用该新功能。

例如 `会员 A 命中 new_recharge_page`，表示会员 A 进入充值页时应展示新版充值页。

### 3.7 未命中灰度

未命中灰度是指会员不满足灰度功能的投放规则，或该功能处于关闭、未生效、已归档状态。

未命中时：

```text
会员端展示旧版
不展示新版入口
不走新版流程
```

### 3.8 白名单

白名单是指定会员直接命中灰度功能的配置。白名单会员不需要再判断 VIP、标签、注册时间、百分比等普通规则。

适用场景：内部测试账号、运营账号、产品账号、指定真实会员、老板账号。

### 3.9 普通规则

普通规则是除白名单以外的投放条件，包括 VIP 等级、会员标签、注册时间段、百分比放量。

第一期普通规则采用"全部满足（AND）"逻辑。例如配置"VIP3 以上 + 最近 30 天注册 + 百分比放量 20%"，则会员必须同时满足全部条件才算命中。

> **补全说明（百分比作用域，详见 6.6）：** 普通规则的判断顺序为"先硬条件准入，再百分比放量"。即百分比放量作用于"已满足 VIP / 标签 / 注册时间 等硬条件的人群"，而不是全站会员。实际命中人数 ≈ 满足硬条件的会员数 × 放量百分比。

### 3.10 百分比放量

百分比放量是指按一定比例开放新功能，如 10% / 30% / 50% / 100%。

百分比放量需保证同一会员结果稳定：

```text
会员今天命中新版，明天也应该命中新版
不能刷新一次新版，再刷新一次旧版
```

> **补全说明（hash 规则，详见 6.5）：** 分流必须按 `hash(memberId + "_" + featureKey) % 100` 计算 slot（0-99），命中条件统一为 `slot < 放量百分比`。
> - **加盐**：拼接 featureKey，保证各功能独立分流，避免多个功能命中同一批会员。
> - **递增稳定**：因使用 `slot < 阈值`，放量从 10% 提升到 30% 时，原 slot 0-9 仍命中，老灰度用户不丢失。

### 3.11 推全量

推全量是指某个灰度功能验证通过后对全部会员开放。推全量后：所有会员默认命中、不再判断投放规则、会员端统一展示新版。单个功能推全量不影响其他灰度功能。

### 3.12 关闭 / 回滚

关闭或回滚是指停止某个灰度功能的投放，所有会员恢复旧版。关闭后所有会员不再命中、统一展示旧版。

关闭只影响功能展示或新版流程，**不删除灰度期间已经产生的正式数据**。例如新版充值页关闭后，之前通过新版充值页产生的充值订单仍然保留。

### 3.13 数据互通

数据互通是指灰度用户和非灰度用户使用同一套正式数据。灰度期间产生的充值订单、投注订单、账变记录、活动领取记录、用户行为日志等，都进入正式数据表。

本期不做：数据库隔离、会员隔离、订单隔离、账变隔离、灰度环境隔离。

### 3.14 人群重叠

人群重叠是指同一个会员同时命中多个灰度功能。处理原则：

```text
不同模块功能允许重叠
同模块但不冲突功能允许重叠
同页面、同入口、同流程的功能需要做互斥处理
```

### 3.15 互斥功能

互斥功能是指多个灰度功能控制同一个页面、同一个入口或同一套业务流程，不能让同一个会员同时命中。例如 `recharge_page_v2` 与 `recharge_page_v3` 都控制充值页主流程，需配置互斥组。

### 3.16 互斥组

互斥组用于解决多个灰度功能冲突问题。同一互斥组内，如果会员同时命中多个功能，后端只返回优先级最高的一个。

| 功能 Key | 功能名称 | 互斥组 | 优先级 |
| ---- | ---- | ---- | ---- |
| recharge_page_v2 | 充值页 V2 | recharge_page | 1 |
| recharge_page_v3 | 充值页 V3 | recharge_page | 2 |

会员同时命中 V2、V3 时，最终返回 `recharge_page_v3`。

### 3.17 优先级

优先级用于互斥组内决定最终命中哪个灰度功能。规则：**数字越大，优先级越高**。

### 3.18 灰度状态

灰度状态用于表示当前灰度功能的生命周期。

| 状态 | 说明 |
| ---- | ---- |
| 未生效 | 当前时间未到生效时间，所有会员不命中 |
| 灰度中 | 按投放规则判断会员是否命中 |
| 已全量 | 所有会员默认命中 |
| 已关闭 | 所有会员不命中，展示旧版 |
| 已归档 | 历史配置，不再参与灰度判断 |

> **补全说明（状态流转，详见第七章）：** 状态之间支持反向流转，包括 已全量 → 灰度中（回退）、已全量 → 已关闭（全量后故障直接关闭）、已关闭 → 灰度中（修复后重开）。完整流转表见 7.6。

### 3.19 操作日志

操作日志用于记录后台对灰度功能的关键操作，包括新建、编辑投放规则、推送全量、关闭回滚、归档配置。用于后续排查"谁在什么时候改了什么"。

### 3.20 前端灰度判断

前端灰度判断是指前端根据后端返回的功能 Key 决定展示新版还是旧版。

```javascript
isGrayFeature("new_recharge_page")
// true  → 展示新版充值页
// false → 展示旧版充值页
```

前端不处理复杂规则，也不处理互斥冲突，只使用后端返回的最终结果。

## 六、投放规则需求

### 6.1 指定会员白名单

后台支持配置会员白名单。白名单会员直接命中该灰度功能，不需要再判断 VIP、标签、注册时间、百分比等普通规则。

支持方式：手动输入会员 ID、批量粘贴会员 ID、批量导入会员 ID。

适用场景：内部测试账号、运营账号、指定真实会员、指定代理下会员。

### 6.2 VIP 等级

支持按会员 VIP 等级投放，如 VIP1-VIP3 可见、VIP5 以上可见、指定 VIP 等级可见。

### 6.3 会员标签

支持按会员标签投放，如新用户、活跃用户、高价值用户、沉默用户、首充用户、代理用户、风险用户。会员命中所选标签后，进入后续规则判断。

### 6.4 注册时间段

支持按会员注册时间筛选，如指定日期区间、最近 7 天注册、最近 30 天注册。

### 6.5 百分比放量

支持按比例放量（5% / 10% / 30% / 50% / 100%）。

**分流规则（补全）：**

```text
slot = hash(memberId + "_" + featureKey) % 100   // 取值 0 - 99
命中条件：slot < 放量百分比
```

要求：

```text
1. 加盐：必须拼接 featureKey 作为盐值，保证每个功能独立分流，
   避免多个功能命中同一批会员、灰度样本偏斜。
2. 一致性：hash 建议使用跨语言一致算法（CRC32 / MurmurHash），
   前后端及多服务结果必须一致、分布均匀。
3. 稳定性：同一会员、同一功能、规则不变时 slot 恒定，
   不允许刷新一次新版、刷新一次旧版。
4. 递增稳定：放量从 10% → 30% 时（slot 0-9 → 0-29），
   原命中会员仍命中，不被甩出。
5. 缩量提示：降低放量比例会剔除部分会员，后台需二次确认。
```

示例：

```text
会员 ID = 100001，功能 Key = new_recharge_page
slot = hash("100001_new_recharge_page") % 100 = 7
放量 10% → 7 < 10 → 命中
放量 30% → 7 < 30 → 仍命中
```

### 6.6 普通规则叠加逻辑

除白名单外，普通规则第一期采用"全部满足（AND）"逻辑。

**判断流水线（补全，明确百分比作用域）：**

```text
第 1 步：硬条件准入（全部满足 / AND）
        - VIP 等级
        - 会员标签
        - 注册时间段
        不满足 → 直接未命中，不进入百分比判断

第 2 步：在通过第 1 步的会员中，按 hash 做百分比放量
        slot < 放量百分比 → 命中
```

**口径声明（供运营理解）：**

```text
实际命中人数 ≈ 满足硬条件的会员数 × 放量百分比
百分比不是"全站会员的百分比"，而是"目标人群内的百分比"
```

示例：

```text
配置：VIP3 以上 + 最近 30 天注册 + 20% 放量
全站会员 100 万；其中满足硬条件 = 5 万
最终命中 ≈ 5 万 × 20% = 1 万
```

后续可扩展为"满足任一条件 / 自定义规则组合"，第一期不做。

## 七、灰度状态与操作

### 7.1 未生效

当前时间未到生效时间时，功能状态为未生效。所有会员不命中，会员端展示旧版。

### 7.2 灰度中

系统根据投放规则判断会员是否命中。命中展示新版，未命中展示旧版。

### 7.3 推全量

后台支持将单个灰度功能推送全量。推全量后该功能对全部会员开放、不再判断投放规则、状态变为已全量。A 功能推全量不影响 B 功能继续灰度。推全量操作需二次确认。

### 7.4 关闭 / 回滚

后台支持关闭单个灰度功能。关闭后全部会员不再命中、统一展示旧版、状态变为已关闭。关闭只影响功能入口、页面展示或新版逻辑，不删除灰度期间已产生的正式业务数据。关闭操作需二次确认。

### 7.5 归档

对已结束、不再使用的灰度配置可进行归档。归档后默认不在列表展示，可通过筛选查看历史记录，不影响已产生的数据和操作日志。第一期不做物理删除。

### 7.6 状态流转规则（补全）

状态之间支持以下流转，含反向流转：

| 当前状态 | 可流转到 | 触发方式 | 说明 |
| ---- | ---- | ---- | ---- |
| 未生效 | 灰度中 | 系统自动 | 到达生效时间 |
| 未生效 | 已关闭 | 手动 | 生效前取消 |
| 灰度中 | 已全量 | 手动（推全量，二次确认） | 全部会员命中 |
| 灰度中 | 已关闭 | 手动（关闭，二次确认） | 全部会员回旧版 |
| **已全量** | **灰度中** | 手动（回退到灰度，二次确认） | 全量后发现局部问题，缩回灰度 |
| **已全量** | **已关闭** | 手动（关闭，二次确认） | 全量后严重故障，直接全员回旧版 |
| **已关闭** | **灰度中** | 手动（重新开启） | 修复后再次灰度 |
| 灰度中 / 已全量 / 已关闭 | 已归档 | 手动（归档） | 不再参与灰度判断 |
| 已归档 | 已关闭 | 手动（恢复，第一期可不做） | 误归档时恢复 |

**状态流转图：**

```text
              ┌──────────┐
              │  未生效   │
              └────┬─────┘
        到达生效时间 │     │ 取消
                  ↓     ↓
              ┌──────────┐  关闭   ┌──────────┐
   重新开启 ┌─→│  灰度中   │────────→│  已关闭   │
          │  └──┬────┬──┘         └────┬─────┘
          │  推全量│    │回退到灰度        │ 归档
          │     ↓    │                  ↓
          │  ┌──────────┐  关闭      ┌──────────┐
          └──┤  已全量   │──────────→ │  已归档   │
             └────┬─────┘   归档      └──────────┘
                  └────────────────────────↑
```

**已全量 → 已归档 的语义：**

```text
归档代表彻底结束灰度生命周期，后端不再做灰度判断。
若功能已全量后归档，意味着"新版成为唯一版本"：
  - 需配合前端 / 代码层把新版设为默认逻辑
  - 旧版代码可下线
  - 归档前确认前端已不再依赖该功能 Key 做新旧判断
```

## 八、灰度人群重叠处理

### 8.1 基本原则

多个灰度功能可以同时存在，投放人群允许重叠，同一会员可以同时命中多个灰度功能。

接口返回示例：

```json
{
  "grayFeatures": [
    "new_recharge_page",
    "new_activity_center",
    "new_user_center"
  ]
}
```

### 8.2 不同模块功能重叠

不同模块的灰度功能允许同时命中，各模块独立判断，互不影响。

### 8.3 同一模块但不冲突功能重叠

同一模块内控制不同功能点的灰度功能允许同时命中，前端按功能 Key 分别判断。例如充值模块内 `new_recharge_page`、`new_recharge_amount_button`、`new_recharge_channel_sort` 可同时命中。

### 8.4 同一模块且互斥功能重叠

两个灰度功能控制同一位置 / 入口 / 流程时，不能让会员同时命中多个版本。例如 `recharge_page_v2` 与 `recharge_page_v3`，需配置互斥组和优先级。

## 九、互斥组需求

### 9.1 后台配置字段

| 字段 | 是否必填 | 说明 |
| ---- | ---- | ---- |
| 是否互斥 | 否 | 默认否 |
| 互斥组 | 否 | 同一组内的灰度功能不能同时命中 |
| 优先级 | 否 | 同一互斥组内用于决定最终命中哪个功能 |

### 9.2 互斥组裁决规则（补全：含白名单分层）

同一互斥组内若会员命中多个功能，按"命中方式分层 + 组内优先级"裁决：

```text
1. 收集该会员在本互斥组内命中的所有功能
2. 若组内存在【白名单命中】的功能：
     仅在白名单命中的功能中，按 priority 取最高，返回
3. 若组内不存在白名单命中：
     在【普通规则命中】的功能中，按 priority 取最高，返回
```

含义：

```text
白名单命中 > 普通规则命中（命中方式优先）
同一命中方式内部，再看 priority 数字（越大越高）
```

**示例（白名单 override）：**

| 功能 Key | 互斥组 | priority | 命中方式 |
| ---- | ---- | ---- | ---- |
| recharge_page_v2 | recharge_page | 1 | 白名单命中 |
| recharge_page_v3 | recharge_page | 2 | 普通规则命中 |

```text
裁决结果：返回 recharge_page_v2
原因：组内存在白名单命中，只在白名单命中范围内比优先级，
      测试账号被强制锁定到 V2，符合白名单本意。
```

**示例（普通命中场景）：** 若 V2、V3 均为普通规则命中，则返回 priority 更高的 V3。

### 9.3 后端处理方式

后端先计算会员命中的所有灰度功能，再处理互斥组。

处理前：

```json
{ "grayFeatures": ["recharge_page_v2", "recharge_page_v3", "new_activity_center"] }
```

处理后（V2、V3 同组，按 9.2 裁决保留一个）：

```json
{ "grayFeatures": ["recharge_page_v3", "new_activity_center"] }
```

### 9.4 前端处理方式

前端不处理互斥冲突，只根据后端最终返回的功能 Key 列表判断展示新版或旧版。冲突由后端提前处理好。

## 十、后端需求

### 10.1 灰度配置管理

后端支持创建灰度功能、编辑灰度规则、查询灰度列表、查询灰度详情、推送全量、关闭回滚、归档配置。

### 10.2 规则匹配能力

会员请求灰度功能列表时，后端根据会员属性判断是否命中。判断维度：会员 ID、VIP 等级、会员标签、注册时间、百分比放量、功能状态、生效时间、结束时间、互斥组、优先级。

### 10.3 命中判断顺序（补全：含白名单优先级与百分比作用域）

```text
1. 判断功能状态
   - 已关闭 / 已归档 / 未生效：全部不命中
   - 已全量：全部命中

2. 判断白名单
   - 白名单会员直接命中，标记命中方式 = 白名单

3. 判断普通投放规则（按 6.6 流水线）
   - 第 1 步 硬条件准入：VIP 等级 / 会员标签 / 注册时间（AND）
   - 第 2 步 百分比放量：在通过硬条件的会员中按
     hash(memberId + "_" + featureKey) % 100 < 放量百分比 判断
   - 命中则标记命中方式 = 普通规则

4. 得出当前会员命中的全部功能 Key（含命中方式标记）

5. 判断是否存在互斥组冲突

6. 互斥组裁决（按 9.2）
   - 组内存在白名单命中 → 仅在白名单命中中按 priority 取最高
   - 否则 → 在普通命中中按 priority 取最高

7. 返回最终命中的功能 Key 列表
```

### 10.4 规则异常处理

如果灰度规则配置异常，为保证线上稳定，默认不命中灰度：展示旧版、记录异常日志、不影响会员正常访问。

### 10.5 性能与缓存建议

```text
该接口位于登录及关键页面前置链路，建议：
- 灰度功能配置在后端做内存 / 本地缓存，配置变更时刷新
- 会员命中结果可按会员维度做短时缓存（与前端 TTL 配合）
- 规则匹配避免逐条 DB 查询，优先批量加载生效中功能配置
```

## 十一、会员端需求

### 11.1 获取命中的灰度功能

会员登录后，前端请求接口获取当前会员命中的灰度功能 Key 列表。

```json
{ "grayFeatures": ["new_recharge_page", "new_activity_center"] }
```

### 11.2 前端通用判断方法

前端封装通用判断方法 `isGrayFeature("new_recharge_page")`，命中展示新版、未命中展示旧版。各业务模块按需接入，不重复实现判断逻辑。

### 11.3 灰度结果缓存与刷新（补全：明确刷新机制）

前端可缓存灰度结果，但需支持重新拉取，避免后台已关闭 / 推全量后仍用旧缓存。

**机制：全局配置版本号 + TTL 兜底（第一期方案）**

1. **后端维护全局版本号 `grayConfigVersion`**

```text
任意灰度功能配置变更（新建 / 编辑规则 / 改状态 / 推全量 / 关闭 / 归档）→ 版本号 +1
```

2. **前端比对版本号**

```text
前端缓存 grayFeatures 列表 + 本地 localVersion
在以下时机携带 localVersion 校验：
  - 页面切换 / 路由跳转
  - 进入资金类核心页面前（充值、提现、投注确认）
  - 普通接口响应头顺带返回最新 grayConfigVersion
localVersion != 服务端版本 → 重新拉取并更新缓存
```

3. **TTL 兜底**

```text
本地缓存 TTL ≤ 5 分钟，超时强制重拉
保证后台关闭 / 推全量后最长 5 分钟内全端生效
```

4. **强制刷新场景（无视缓存）**

```text
会员登录后 / 切换账号后 / 手动下拉刷新 / 页面硬刷新
```

5. **高危变更增强（可选，第一期可不做）**

```text
推全量、关闭可叠加服务端推送（WebSocket / Long Polling）即时下发
第一期"版本号 + TTL≤5min"即可满足兜底
```

### 11.4 未登录用户处理

第一期未登录用户默认不参与灰度：不请求会员灰度功能列表、默认展示旧版、登录后重新获取。后续如需未登录灰度，可扩展设备 ID / 浏览器 ID 分流。

## 十三、数据互通与回滚要求（补全：版本标识与兼容）

由于本次灰度不做环境隔离，灰度用户产生的数据就是正式数据。涉及充值、提现、投注、结算、派奖、返水、红利、账变、活动领取等资金链路时，需满足以下要求。

### 13.1 核心数据表增加来源标识字段

在涉及资金 / 订单 / 账变的核心表（充值订单、投注订单、账变记录、活动领取记录等）增加字段：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| source_feature_key | varchar，可空 | 产生该数据的灰度功能 Key，如 `new_recharge_page`；旧版 / 全量可为空或填默认值 |
| source_version | varchar，可空 | 版本标识，如 v1 / v2 / v3，区分多版本灰度 |

### 13.2 数据不可篡改

```text
关闭 / 回滚 / 归档灰度功能时，不修改历史数据的 source_feature_key
保证任何时候都能追溯：这笔订单由哪个版本流程产生
```

### 13.3 新旧逻辑字段兼容

```text
新版新增的数据字段必须允许为空（NULL），旧版读取时不报错
旧版必填字段，新版必须继续写入，避免回滚后旧版读到空值
新旧版本写入同一张正式表，结构保持前向 / 后向兼容
```

### 13.4 回滚要求

```text
关闭灰度后历史数据仍可查询
关闭灰度后订单和账变仍可正常处理
关闭灰度不影响已产生的充值、投注、活动记录
不会因功能回滚导致资金数据异常
```

### 13.5 用途与对账价值

```text
回滚排查：按 source_feature_key 定位故障版本产生的数据
对账兼容：财务对账可识别不同版本订单
灰度评估：可按版本统计转化率 / 客诉率 / 异常率，作为是否推全量的决策依据
```

示例：

```text
新版充值页产生的订单 source_feature_key = "new_recharge_page"
关闭新版充值页后该字段保持不变，后台仍可正常查询、处理、对账该订单
新订单走旧版逻辑，source_feature_key 为空
```

## 十七、流程说明

### 17.1 灰度命中流程

```text
会员登录
↓
前端请求灰度功能接口（携带本地 grayConfigVersion）
↓
后端获取会员属性
↓
后端按命中顺序匹配灰度规则（状态→白名单→硬条件→百分比）
↓
后端处理互斥组冲突（白名单分层 + 优先级）
↓
返回最终命中的功能 Key 列表 + 最新 grayConfigVersion
↓
前端根据功能 Key 判断展示新版或旧版
```

### 17.2 推全量流程

```text
后台点击推全量 → 二次确认 → 状态改为已全量 → grayConfigVersion +1
→ 所有会员默认命中 → 前端重新拉取后展示新版
```

### 17.3 关闭回滚流程

```text
后台点击关闭 → 二次确认 → 状态改为已关闭 → grayConfigVersion +1
→ 所有会员不再命中 → 前端重新拉取后展示旧版
```

### 17.4 全量回退流程（补全）

```text
后台对已全量功能点击回退 → 选择回退到"灰度中"或"已关闭" → 二次确认
→ 状态变更 → grayConfigVersion +1 → 前端重新拉取后按新状态展示
→ 历史已产生数据的 source_feature_key 保持不变
```

## 十八、验收标准

### 18.1 后台验收

1. 可创建多个灰度功能，每个有唯一功能 Key，Key 不允许重复。
2. 支持所属模块、状态、生效时间、投放规则配置。
3. 支持白名单、VIP 等级、会员标签、注册时间段、百分比放量配置。
4. 支持互斥组及优先级配置，禁止同组相同优先级。
5. 支持单个功能推全量、关闭回滚、归档。
6. **支持已全量回退到灰度 / 关闭、已关闭重新开启等反向流转。**
7. 所有关键操作（含反向流转）都有操作日志。

### 18.2 后端验收

1. 会员登录后可获取命中的灰度功能 Key 列表。
2. 白名单会员可直接命中。
3. 普通规则按"硬条件 AND + 百分比"判断，**百分比作用于满足硬条件的人群**。
4. **百分比分流使用 `hash(memberId + featureKey) % 100`，加盐独立、对同一会员稳定、放量提升不剔除原命中会员。**
5. 已关闭 / 已归档 / 未生效全部不命中，已全量全部命中。
6. 多个不冲突功能可同时命中。
7. **同一互斥组内命中多个功能时，白名单命中优先；同层按优先级取最高。**
8. 规则异常默认不命中，不影响主流程。
9. 灰度关闭后不影响已产生的正式业务数据。
10. **资金链路数据写入 source_feature_key，回滚后字段不变、可追溯。**

### 18.3 前端验收

1. 可获取当前会员命中的灰度功能 Key 列表，封装统一判断方法。
2. 业务页面按功能 Key 展示新版 / 旧版。
3. 不自行处理互斥冲突，只用后端最终结果。
4. 切换账号 / 页面刷新后重新获取。
5. **配置版本号变化或 TTL 超时后能重新拉取，后台推全量 / 关闭 / 回退后正确展示。**
6. 未登录用户默认展示旧版。

## 十九、测试用例简表

| 场景 | 操作 | 预期结果 |
| ---- | ---- | ---- |
| 新建灰度功能 | 填写 Key、名称、模块、规则后保存 | 创建成功 |
| 功能 Key 重复 | 使用已有 Key 创建 | 保存失败 |
| 白名单命中 | 会员在白名单中 | 直接命中新功能 |
| VIP 命中 | 会员满足 VIP 规则 | 命中新功能 |
| 标签命中 | 会员拥有指定标签 | 命中新功能 |
| 注册时间命中 | 会员注册时间在范围内 | 命中新功能 |
| 百分比命中 | 会员落入放量比例 | 命中新功能 |
| 百分比未命中 | 会员未落入放量比例 | 展示旧版 |
| **百分比作用域** | 配置硬条件 + 20%，统计命中数 | ≈ 满足硬条件人数 × 20% |
| **百分比放量递增** | 10% 命中后调到 30% | 原命中会员仍命中 |
| **分流稳定性** | 同一会员多次刷新 | 结果一致，不来回切换 |
| 多功能命中 | 同时命中多个不冲突功能 | 返回多个功能 Key |
| 互斥组冲突（普通） | 同时命中同组多个普通功能 | 只返回优先级最高 Key |
| **互斥组冲突（白名单）** | 白名单命中 V2、普通命中 V3 | 返回 V2（白名单层优先） |
| 推全量 | 后台点击推全量 | 所有会员命中新功能 |
| 关闭回滚 | 后台点击关闭 | 所有会员展示旧版 |
| **全量回退** | 已全量功能回退到灰度 / 关闭 | 按新状态展示，历史数据 Key 不变 |
| **缓存刷新** | 后台关闭后，前端 5 分钟内 | 重新拉取并展示旧版 |
| 未登录访问 | 未登录会员进入页面 | 默认展示旧版 |
| 规则异常 | 规则配置异常 | 默认不命中，展示旧版 |
| **资金数据追溯** | 新版充值产单后关闭功能 | 订单 source_feature_key 不变，可查可处理 |
| 操作日志 | 编辑、推全量、关闭、回退 | 日志正常记录 |

---

## 二十、风险说明

1. 灰度数据与正式数据互通，涉及资金链路时必须确认新旧逻辑兼容，并写入 source_feature_key 保证可追溯。
2. 同一页面存在多个新版方案时必须配置互斥组，避免会员同时命中多个版本；白名单命中享有更高裁决优先级。
3. 功能 Key 一旦接入前端不建议随意修改或删除。
4. 推全量、关闭回滚、全量回退属高风险操作，需二次确认和权限控制。
5. 前端缓存需配合全局版本号 + TTL，避免后台变更后不生效。
6. 灰度功能关闭后不应影响已产生的正式业务数据。
7. 互斥组优先级配置错误可能导致会员看到错误版本，后台保存时禁止同组相同优先级。
8. 百分比放量必须固定 hash 分流并加盐，否则会员刷新后新旧版来回切换、或多功能命中同一批人导致样本偏斜。

## 二十一、简版总结

灰度发布功能用于控制新功能逐步开放。后台负责创建灰度功能、配置投放规则、处理状态（含全量回退等反向流转）和操作日志；后端负责根据会员属性按"状态 → 白名单 → 硬条件 → 百分比"顺序计算命中结果，并按"白名单分层 + 优先级"处理互斥冲突；前端负责根据后端返回的功能 Key 展示新版或旧版，并通过全局版本号 + TTL 保证缓存及时刷新。

本次灰度是功能级灰度，不做环境隔离。多个功能可同时灰度，会员可同时命中多个不冲突功能。同一页面 / 流程的多个版本通过互斥组和优先级保证最终只命中一个版本。涉及资金链路的数据通过 source_feature_key 保证回滚后可追溯、可兼容。
