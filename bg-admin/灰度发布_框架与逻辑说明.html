这是什么 / 边界

一句话：给单个新功能做开关式灰度——先放给一部分会员用，验证没问题再推全量，出问题可一键回滚。

✅ 灰度控制什么
  • 功能是否展示
  • 入口是否开放
  • 走新版还是旧版逻辑
⛔ 明确不做
  • 环境隔离 / 数据库隔离
  • 会员隔离 / 订单隔离
  • APP 版本控制 / 复杂规则表达式
🔑 设计前提
“不做环境隔离、灰度数据即正式数据” 是整套设计的根。正因为如此，资金链路才需要「来源标识」字段（见第 6 节），用来回滚后追溯。

3

核心逻辑：会员怎么算「命中」

后端对每个灰度功能，按固定顺序判断当前会员是否命中。顺序不能乱。

先看功能状态
不命中已关闭 / 已归档 / 未生效  ·  全命中已全量  ·  继续判断灰度中
白名单优先
会员在白名单里 → 直接命中，标记「白名单命中」，不再看普通规则。
普通规则（两步，顺序不能反）
① 硬条件准入（AND，必须全满足）：VIP 等级 + 会员标签 + 注册时间段
② 百分比放量：在「通过①的人」里，按 hash 分流取一定比例 → 命中标记「普通命中」
收集 + 互斥裁决
汇总该会员命中的所有功能 Key（带命中方式标记）→ 处理互斥组（见第 4 节）。
返回最终命中的功能 Key 列表
前端拿这个列表展示新版。
📊 运营要理解的口径
百分比是「目标人群内的百分比」，不是全站。
例：VIP3以上 + 近30天注册 + 20% → 命中数 ≈ 满足前两个条件的人数 × 20%。
🔒 分流要稳定
同一会员同一功能结果恒定——不能刷新一下新版、刷新一下旧版。
📈 放量不掉人
从 10% 提到 30%，原来命中的人继续命中，不被甩出。
实现：hash(会员ID + "_" + 功能Key) % 100 < 放量百分比 （拼功能Key做"盐"，保证各功能独立分流，互不串号）
4

互斥与重叠

✅ 允许重叠
一个会员可同时命中多个不冲突的功能（如 新版充值页 + 新版活动中心），各自独立判断。
⛔ 必须互斥
多个版本控制同一页面/入口/流程（如充值页 V2 与 V3），不能让同一会员同时命中两个。

互斥裁决规则（配到同一「互斥组」，后端只保留一个）：

同一互斥组内命中多个时： ① 先看命中方式：白名单命中 > 普通命中 ② 同一命中方式内：优先级数字越大越优先
💡 举例
测试号被白名单锁到 V2、同时普通命中 V3 → 返回 V2。白名单本意就是把测试号固定到指定版本。
前端不碰互斥，后端裁决完只给前端一个干净的 Key 列表。

7

⚠️ 待开发确认项（沟通重点）

工作量大、跨团队、或方案有多选的点。建议对每项让开发给「能做 / 要改方案 / 工作量」三选一，避免来回拉扯。

1hash 分流算法 · 跨语言一致
前后端语言不同，要保证同一会员算出同一 slot，否则新旧版来回跳。
倾向：统一选 CRC32 或 MurmurHash；由谁提供一致实现待定。
2核心资金表加「来源标识」字段
要动充值/投注/账变等核心表结构，涉及历史数据兼容、对账、DBA —— 最容易被顶回来的点。
倾向：新字段必须可空、向前兼容；全表加 or 只加关键表待定。
3全局版本号 + TTL 缓存刷新
涉及前端架构改造和接口埋版本号，工作量集中在前端。
倾向：第一期「版本号 + TTL≤5min」；WebSocket 即时下发列为二期。
4灰度接口在登录关键链路上
接口慢会拖累登录/核心页，需后端做缓存方案。
倾向：配置内存缓存 + 会员命中结果短时缓存，方案待后端定。
5百分比「缩量」的处理
放量比例调低会把部分会员甩回旧版，属敏感操作。
倾向：后台二次确认 + 明确提示影响范围。

第一期范围（防蔓延）

✅ 第一期做
  • 功能级灰度
  • 5 类投放规则：白名单 / VIP / 标签 / 注册时间 / 百分比
  • 互斥组 + 优先级
  • 状态流转（含反向回退）
  • 推全量 / 关闭 / 归档
  • 资金来源标识 + 版本号缓存
⛔ 第一期不做
  • 环境 / 数据隔离
  • 未登录用户灰度
  • 复杂规则表达式（OR / 自定义组合）
  • 灰度数据单独结算
  • WebSocket 即时下发
  • 误归档恢复
